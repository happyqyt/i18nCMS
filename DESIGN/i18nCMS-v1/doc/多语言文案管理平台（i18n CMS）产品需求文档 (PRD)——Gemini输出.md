## 1. 项目概述

### 1.1 项目背景
随着企业全球化业务拓展，传统基于 Excel 或分散文件的文案管理模式面临管理混乱、协作低效、翻译成本高及发布周期长等痛点。本系统旨在建立一个集中化、流程化、智能化的平台，覆盖文案从创建、翻译、审核到发布的全生命周期。

### 1.2 建设目标
* **集中管理**：打破数据孤岛，统一项目文案存储。
* **快速交付**：支持多格式导出及一键发布至 S3。
* **资产复用**：通过公共文案库和统计功能提升复用率，降低重复翻译成本。

## 3. 功能需求详述

### 3.1 目录管理（树形结构）
系统采用树状层级组织文案，支持灵活的层级配置。

* **新增目录**：
    * 层级：支持无限层级。
    * 配置：需指定**审核员**（多个）、**层级Key**（导出JSON时的层级对象名，默认 `layout_{id}`）。
    * 生效规则：一级目录由任何人创建后即刻生效；子目录需父级审核。
* **编辑与迁移**：支持修改目录名、审核员及父级目录变更（需审核）。
* **删除规则**：执行软删除。删除请求需经审核通过后，该目录及其下属子目录、文案执行物理删除（硬删除）。

### 3.2 文案管理

#### 3.2.1 文案 CRUD
* **录入模式**：
    * 单行模式：多语种并列编辑（适用于多语言项目）。
    * 多行模式：仅编辑源语言（适用于项目初期或单语言项目）。
* **富文本支持**：编辑器支持变量占位符（如 `{user_name}`）及 HTML 标签。
* **删除逻辑**：支持单条或批量软删除，亦支持按特定语种维度删除翻译内容。

#### 3.2.2 Key 生成与管理规则
* **自动生成**：截取原文案前16字符，空格转下划线 `_`，保留大小写。
* **冲突处理**：若 Key 重复，自动拼接序列号（如 `key_1`, `key_2`）。
* **唯一性范围**：**建议全局唯一或顶层目录下唯一**（确保导出多层级 JSON 时不会产生逻辑覆盖）。

#### 3.2.3 导入与导出
* **智能导入**：
    * 支持格式：JSON, Excel, CSV。
    * 层级识别：导入多层级 JSON 时自动创建对应目录树。
    * 冲突检测：同项目内 Key 重复需弹窗提示。
* **导出与发布**：
    * **发布至 S3**：仅支持 JSON 格式，直接上传至 S3 指定路径。
    * **前置检查**：检查所选语言翻译是否完整。若存在“草稿”状态的文案，需二次确认（发布内容不包含草稿）。
    * **链接关联**：发布成功后生成 S3 访问链接并关联至对应目录。

#### 3.2.4 资产复用与统计
* **查重统计**：以英语为基准，统计全系统内内容相同的文案。
* **一键同步**：在统计视图下，可将某一 Key 下的翻译内容一键同步至其他相同原文的 Key。
* **生效逻辑**：同步操作生成的修改统一进入“草稿”状态，需审核。

## 4. 流程逻辑

### 4.1 核心审核流（草稿制）
为保证线上文案稳定性，引入**操作即草稿**机制：
1.  **操作触发**：任何人的增删改操作（非一级目录创建）均先存为草稿，仅本人可见。
2.  **逐层审核**：操作需经过**所有父级目录**定义的审核员逐层审批。
3.  **互斥审核**：审核员本人的操作需由该目录下的其他审核员审批。
4.  **生效**：审批完成后，文案对全员可见并进入可发布版本。

### 4.2 版本管理
* **打标**：每次执行“发布”操作时，系统自动对该目录节点记录一个版本号。
* **版本比对**：支持 Diff 视图，高亮展示两个版本间的文案差异。
* **回滚**：支持一键回滚目录至历史指定版本。

## 5. 系统设置与扩展

### 5.1 语种管理
* **标准规范**：语言码遵循系统预设 List（参考 VeSync Wiki 规范）。
* **校验机制**：导入时强制校验语言码合法性。

### 5.2 迭代优化（Roadmap）
* **AI 翻译集成**：接入 Google、DeepL、阿里云等 API，实现一键预翻译。
* **开发者工具**：提供 CLI 工具或插件，支持开发环境下自动拉取最新文案。

## 6. 待确认问题 (Open Issues)
1.  **删除逻辑确认**：软删除申请经审核后，是执行硬删除（彻底抹除数据）还是仅更改状态位？（建议保留历史记录以备审计）。
2.  **Key 的唯一性**：确定 Key 是同级唯一还是项目内全局唯一？（建议项目内唯一以简化前端调用逻辑）。
3.  **审核通知**：是否需要集成钉钉/飞书通知提醒审核员？
